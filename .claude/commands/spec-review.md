# /spec-review — 规范合规审查

> 来源：Superpowers subagent-driven-development / spec-reviewer-prompt.md
> 适配：Hermes 调度 CC 执行

## 触发时机

CC 任务完成后，**必须先通过规范合规审查，才能进入代码质量审查**。

顺序不可颠倒：
```
CC 实现 → /spec-review（规范合规） → /code-quality-review（代码质量） → Hermes 复核
```

## 前置条件

CC 必须已完成以下工作并报告：
- 改动了哪些文件
- 每个文件的核心逻辑
- 声称实现的功能列表

## 执行步骤

### 1. 确认任务要求

Hermes 提供给 CC 的信息（Hermes 准备）：
- 原始任务描述（来自用户指令）
- CC 声称实现的内容

### 2. 读取实现代码

```bash
cd /home/ubuntu/vc-2.0
git diff --stat
git diff {改动的文件路径}
```

### 3. 逐项核查

对每个改动文件，逐一回答：

**缺失检查 — CC 声称实现了，但实际没有：**
- [ ] 每个声称的功能都有实际代码支撑吗？
- [ ] 是否有遗漏的边界情况处理？

**多余检查 — 任务没要求，但 CC 额外做了：**
- [ ] 是否有超出原始任务范围的改动？
- [ ] 是否有"顺手优化"但不属于本次任务的功能？

**理解偏差检查：**
- [ ] CC 是否理解了正确的业务场景？
- [ ] 解决方案是否与任务意图一致？

**VC2.0 规范合规检查：**
- [ ] 是否遵循 `rules/safety.md`？
- [ ] 是否符合 `CLAUDE.md` 中的架构约定？
- [ ] 是否引入新的安全风险（SQL注入、凭证泄露、端口暴露）？

### 4. 输出报告

```
## 规范合规审查报告

### 审查结论
✅ 通过 / ❌ 未通过

### 缺失项（CC 声称做了但实际没有）
-

### 多余项（任务没要求但 CC 额外做了）
-

### 理解偏差
-

### VC2.0 规范问题
-

### 修复要求
（如果有）
```

## 通过标准

所有检查项均为 ✅，无缺失、无多余、无偏差、无规范问题时，才能进入下一阶段。

## 审查失败处理

- Hermes 记录失败原因
- Hermes 将具体问题反馈给 CC（重新调度 CC 修复）
- CC 修复后重新走一遍 spec-review 流程
- **禁止跳过 spec-review 直接进入 code-quality-review**
